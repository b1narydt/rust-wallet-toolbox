**1. test_sync_to_writer_non_empty_remote (PARITY-06):**
- Call `make_wallet_with_remote_backup().await` to get a SetupWallet with local active + remote StorageClient backup
- Get the identity_key: `let ik = identity_key_hex(&test_root_key());`
- Seed 3 outputs into the local store: `common::seed_outputs(&setup.storage, &ik, 3, 1000).await`
- Call `setup.storage.update_backups(None).await.unwrap()` -- this calls sync_to_writer internally for each backup
- Assert `inserts > 0` ("seeded local data must sync to remote: inserts={inserts}")
- This is the key difference from TEST-06: we seed real data before syncing, proving the sync loop processes actual non-empty SyncChunks

**2. test_funded_key_live (PARITY-07):**
- Read `BSV_FUNDED_KEY` from env var (or skip test if not set): `let funded_hex = match std::env::var("BSV_FUNDED_KEY") { Ok(h) => h, Err(_) => { eprintln!("BSV_FUNDED_KEY not set, skipping funded-key test"); return; } };`
- Parse as PrivateKey: `let funded_key = PrivateKey::from_hex(&funded_hex).expect("valid funded key");`
- Build a wallet with this funded key:
  ```
  let setup = WalletBuilder::new()
      .chain(Chain::Test)
      .root_key(funded_key.clone())
      .with_sqlite_memory()
      .with_services(Arc::new(common::MockWalletServices))
      .build()
      .await
      .expect("build funded wallet");
  ```
- Wire a StorageClient as backup (same pattern as make_wallet_with_remote_backup but using funded_key):
  ```
  let proto = WalletArc::new(ProtoWallet::new(funded_key));
  let client = Arc::new(StorageClient::new(proto, STAGING_ENDPOINT)) as Arc<dyn WalletStorageProvider>;
  setup.storage.add_wallet_storage_provider(client).await.expect("add funded-key backup");
  ```
- Verify the funded wallet can interact with the server: `let stores = setup.storage.get_stores().await;`
- Assert `stores.len() >= 2` ("funded wallet has local + remote stores")
- Assert remote store has non-empty `storage_identity_key`
- NOTE: Do NOT attempt create_action or internalizeAction with a BEEF fixture -- the funded key's UTXO state on staging is unknown. The test proves the funded key can authenticate, make_available, and find_or_insert_user on the live server (BRC-31 auth with a different key than BSV_TEST_ROOT_KEY). This is the minimum viable proof for PARITY-07 without risking failures due to missing UTXOs.
- If you want to go further: attempt `setup.storage.update_backups(None).await` and assert it succeeds (proving sync loop works with funded key too).

Both tests must be `#[tokio::test(flavor = "multi_thread")]` and `#[ignore]`.
