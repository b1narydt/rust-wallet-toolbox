# Phase 06.1: Integration Test Parity - Research

**Researched:** 2026-03-25
**Domain:** Rust integration testing — sync protocol, set_active conflict resolution, bidirectional sync, funded-key live tests
**Confidence:** HIGH

## Summary

Phase 06.1 fills integration test gaps identified in a TS parity audit. Phase 06 established the live test infrastructure (6 tests against staging server, all passing). The gaps are in local-storage sync tests — our tests sync empty wallets; the TS suite syncs >1000 records and asserts exact insert/update counts. The other gaps are in manager orchestration: setActive swapping twice, two-wallet isolation, and bidirectional sync.

The funded-key tests (internalizeAction with real BEEF, end-to-end payment) are the highest-risk gap. They require a key that can produce or receive real transactions on the BSV test chain. The provided funded key `19deb354...` is a 64-char hex private key — it can construct a wallet via `WalletBuilder` with a live `StorageClient` as the active store, and that wallet can receive via `internalize_action` if given a valid BEEF. Whether it can `create_action` (send funds) depends on whether there are spendable UTXOs on the test chain for that key.

All new tests belong in `tests/storage_client_integration_tests.rs` (existing file, currently 6 tests). Local-only sync tests (no live server needed) may also go in a new or existing local test file. Separation by requirement:

- **Group A (local SQLite only):** TEST-06.1 through TEST-06.5 — use two in-memory SQLite providers
- **Group B (live staging server):** TEST-06.6 through TEST-06.9 — require BSV_TEST_ROOT_KEY + BSV_FUNDED_KEY env vars

**Primary recommendation:** Implement Group A tests first (no network, no credentials). Group B tests use the funded key from env var and are `#[ignore]`-gated like existing live tests. Do not attempt to derive funded-key BSV balance from thin air — scope the funded-key tests to "wallet receives a payment BEEF" and use a hardcoded BEEF if needed.

### Pattern 1: Seeded Sync Test
**What:** Seed N records into source storage via `seed_outputs` or direct insert, call `sync_to_writer`, assert `inserts > 0`.
**When to use:** TEST-06.1, TEST-06.2, TEST-06.5, TEST-06.6.

```rust
// Source: tests/common/mod.rs seed_outputs + manager.rs sync_to_writer:1375
async fn test_sync_to_writer_with_data() {
    let source = create_provider().await.unwrap();
    let dest = create_provider().await.unwrap();

    let source_settings = source.make_available().await.unwrap();
    let dest_settings = dest.make_available().await.unwrap();
    let (user, _) = source.find_or_insert_user(IDENTITY_KEY).await.unwrap();

    // Seed N records into source
    common::seed_outputs(&*source_manager_wrapper, IDENTITY_KEY, 5, 1000).await;

    let manager = WalletStorageManager::new(IDENTITY_KEY.to_string(), Some(source), vec![]);
    manager.make_available().await.unwrap();

    let (inserts, updates, _log) = manager
        .sync_to_writer(dest.as_ref(), source.as_ref(), &dest_settings, &source_settings, None)
        .await
        .unwrap();

    assert!(inserts > 0, "seeded records must be synced");
    assert_eq!(updates, 0, "first sync should produce only inserts");
}
```

**Critical detail:** `seed_outputs` takes `&WalletStorageManager`, not `&dyn WalletStorageProvider`. For local tests, wrap the `Arc<dyn WalletStorageProvider>` in a minimal manager, OR use the `StorageProvider` methods directly via downcasting. The simpler path: use `WalletStorageManager::new(ik, Some(provider), vec![])` as the source, then seed through the manager.

Actually, looking at `seed_outputs` in `tests/common/mod.rs` line 241, it takes `storage: &WalletStorageManager`. This means we must build a manager and seed through it — consistent with `create_test_wallet()` which already does this pattern.

### Anti-Patterns to Avoid
- **Calling seed_outputs with `Arc<dyn WalletStorageProvider>`:** `seed_outputs` takes `&WalletStorageManager`, not `&dyn WalletStorageProvider`. Build a thin manager wrapper when seeding.
- **Sharing storage across concurrent tests without `--test-threads=1`:** Live tests using the same BSV key will collide on the server's BRC-31 session. Keep `#[ignore]` + run with `--test-threads=1`.
- **asserting `inserts > 1000` for the live test:** The staging server's state for BSV_TEST_ROOT_KEY may not have 1000 records. Assert `inserts > 0` for live tests; exact count assertions only for local seeded tests where you control the data.
- **Using `update_backups` to test `sync_to_writer` directly:** TEST-06.7 must call `sync_to_writer` explicitly, not via `update_backups`, because TEST-06 already covers the `update_backups` path.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Seeding test records | Custom SQL inserts in each test | `common::seed_outputs`, `common::seed_transaction` | Already tested, handles user creation + basket creation + transaction + outputs |
| Two-store setup | Duplicate WalletBuilder calls inline | `create_test_wallet()` / `create_provider()` | Established patterns in common/mod.rs and manager_tests.rs |
| Incremental timestamp ordering | Manually setting chrono times | Let `Utc::now()` drive timestamps + `tokio::time::sleep(Duration::from_millis(10))` between inserts | Ensures `updated_at` ordering is correct for incremental sync |
| Real payment BEEF | Constructing raw transaction hex by hand | Use hardcoded BEEF fixture from TS test (walletLive.man.test.ts line 175) | A valid BEEF is embedded in the TS test as a hex constant; can be reused as a fixture for internalizeAction tests |
| Two distinct test identities | Deriving keys programmatically | Two env vars: `BSV_TEST_ROOT_KEY` + `BSV_TEST_ROOT_KEY_2` | Same pattern as existing single-key env var; operator provides both |

**Key insight:** The TS `Wallet.sync.test.ts` test uses `createLegacyWalletSQLiteCopy` — a pre-seeded SQLite file with >1000 records. We don't have that fixture. The Rust equivalent is to seed N records via `seed_outputs` before asserting. Use N=5 for unit-speed local tests (not 1000 — that's unnecessary for correctness).

## Common Pitfalls

### Pitfall 1: seed_outputs Takes WalletStorageManager, Not WalletStorageProvider
**What goes wrong:** Attempt to call `seed_outputs(Arc<dyn WalletStorageProvider>, ...)` fails at compile time — wrong type.
**Why it happens:** `seed_outputs` signature is `pub async fn seed_outputs(storage: &WalletStorageManager, ...)`.
**How to avoid:** Wrap the provider in a `WalletStorageManager::new(...)` before seeding, then extract the provider reference for `sync_to_writer` calls.
**Warning signs:** Compiler error about `WalletStorageManager` vs `dyn WalletStorageProvider` type mismatch.

### Pitfall 2: sync_to_writer Requires Settings References
**What goes wrong:** `sync_to_writer` takes `writer_settings: &Settings` and `reader_settings: &Settings`. These must be obtained AFTER calling `make_available()` on both providers.
**Why it happens:** Settings are not available until `make_available()` caches them.
**How to avoid:** Always call `make_available()` on both providers, then retrieve settings via `manager.get_active_settings().await`.
**Warning signs:** `Option::unwrap()` panic when settings not cached.

### Pitfall 3: set_active User.active_storage Must Match SIK in Both Stores
**What goes wrong:** Test asserts `user.active_storage == sik_b` in only the active store, but the original store still has `active_storage == sik_a`. The TS test verifies BOTH stores are updated.
**Why it happens:** `set_active` propagates via `sync_to_writer` to all other stores; if this didn't happen, the test wouldn't catch the regression.
**How to avoid:** After `set_active(sik_b)`, directly query `user.active_storage` from BOTH store_a and store_b to verify propagation.
**Warning signs:** Test passes but the user record in the demoted store was never updated.

### Pitfall 4: Incremental Sync Needs Time Gap Between Inserts
**What goes wrong:** Incremental sync uses `since = last_sync_map.max_updated_at`. If the second insert has the same `updated_at` as the first sync's cutoff, the `since` filter may exclude it.
**Why it happens:** SQLite NaiveDateTime has millisecond resolution; if inserts happen in the same millisecond, `>= since` vs `> since` can cause off-by-one.
**How to avoid:** Insert a `tokio::time::sleep(Duration::from_millis(5))` between the initial sync and the second insert, ensuring the second record's `updated_at` is strictly after `since`.
**Warning signs:** Incremental sync returns 0 inserts when 1 is expected.

### Pitfall 5: sync_to_writer Requires User in Reader Store
**What goes wrong:** `sync_to_writer` loop calls `reader.get_sync_chunk(args)` which internally queries the user record. If the user doesn't exist in the reader store, the chunk returns empty with an error.
**Why it happens:** `process_sync_chunk` looks up user by `identity_key` before processing any entities.
**How to avoid:** Always call `reader.find_or_insert_user(IDENTITY_KEY)` before any sync operation.
**Warning signs:** `processSyncChunk: user not found for identity key` error.

### Pitfall 6: Funded Key May Not Have Spendable UTXOs
**What goes wrong:** `create_action` with funded key returns error "insufficient funds" because the staging server has no spendable outputs for that key.
**Why it happens:** The funded key's UTXOs may have been spent or were on a different address derivation path.
**How to avoid:** Scope funded-key tests to `internalize_action` (receive only) using a hardcoded BEEF fixture, not `create_action` (requires balance). The TS test `walletLive.man.test.ts:175` embeds a real BEEF hex constant — use it as an `internalizeAction` fixture.
**Warning signs:** `WERR_INSUFFICIENT_FUNDS` or similar balance error.

### Pitfall 7: TEST-06.7 Direct sync_to_writer Call Requires Explicit Settings References
**What goes wrong:** `sync_to_writer` is called on the manager but requires `&Settings` for both writer and reader. These are stored in `ManagedStorage` and must be extracted.
**Why it happens:** `sync_to_writer` is a free method on the manager that doesn't implicitly use the internal active/backup stores — it takes explicit store references.
**How to avoid:** After `make_available()`, call `manager.get_active_settings().await.unwrap()` for the active settings. For the remote client, call `remote_client.make_available().await.unwrap()` directly to get its settings.
**Warning signs:** Cannot access private `state` field — use the public `get_active_settings()` accessor.

### Seeded Local Sync Test (TEST-06.1 pattern)
```rust
// Source: tests/common/mod.rs:241 (seed_outputs), tests/manager_tests.rs (create_provider)
#[tokio::test]
async fn test_sync_to_writer_with_populated_data() {
    let source_provider = create_provider().await.unwrap();
    let dest_provider = create_provider().await.unwrap();

    source_provider.make_available().await.unwrap();
    dest_provider.make_available().await.unwrap();

    // Build manager wrapping source so we can use seed_outputs
    let source_mgr = WalletStorageManager::new(
        IDENTITY_KEY.to_string(),
        Some(source_provider.clone()),
        vec![],
    );
    source_mgr.make_available().await.unwrap();

    // Seed 5 outputs (also creates user + basket + transaction)
    common::seed_outputs(&source_mgr, IDENTITY_KEY, 5, 1000).await;

    let source_settings = source_mgr.get_active_settings().await.unwrap();
    let dest_settings = dest_provider.make_available().await.unwrap();

    // Sync source -> dest
    let (inserts, updates, _log) = source_mgr
        .sync_to_writer(
            dest_provider.as_ref(),
            source_provider.as_ref(),
            &dest_settings,
            &source_settings,
            None,
        )
        .await
        .unwrap();

    assert!(inserts > 0, "seeded records must transfer: inserts={inserts}");
    assert_eq!(updates, 0, "first sync should have no updates: updates={updates}");

    // Second sync: no changes
    let (inserts2, updates2, _) = source_mgr
        .sync_to_writer(
            dest_provider.as_ref(),
            source_provider.as_ref(),
            &dest_settings,
            &source_settings,
            None,
        )
        .await
        .unwrap();
    assert_eq!(inserts2, 0, "second sync with no new data: inserts={inserts2}");
    assert_eq!(updates2, 0, "second sync with no new data: updates={updates2}");
}
```

## Open Questions

1. **Can the funded key `19deb354...` create_action on staging?**
   - What we know: It's a 64-char hex secp256k1 private key. It can authenticate to the staging server. Whether it has spendable UTXOs on the BSV test chain is unknown without checking the server.
   - What's unclear: UTXO state of the funded key on staging.
   - Recommendation: Scope TEST-06.8 to `internalize_action` receiving a payment, not `create_action` sending one. Use the BEEF fixture from walletLive.man.test.ts (hardcoded BEEF hex) rather than live UTXO spending.

2. **Does insert_output_basket exist on WalletStorageManager for incremental test seeding?**
   - What we know: `seed_outputs` in `tests/common/mod.rs` calls `storage.insert_output_basket(&OutputBasket{...})`. The method exists on manager.
   - What's unclear: Whether we can call it for a single-entity insert in the incremental test without the full seed_outputs overhead.
   - Recommendation: Call `seed_outputs(manager, ik, 1, 1000)` for the incremental insert — simplest, already tested. Even though it inserts tx+basket+output, the sync will report them all as 1+ inserts, which satisfies the "exactly the new entity was synced" criterion.

3. **Two-wallet isolation: same server, different users — can we verify isolation without two funded keys?**
   - What we know: The staging server scopes all data by `userId` derived from `identity_key`. Different keys = different users. The Rust `find_or_insert_user` returns per-key users.
   - What's unclear: Whether a second test key would be rate-limited or rejected on staging.
   - Recommendation: For TEST-06.4 (local two-wallet isolation), use two in-memory SQLite managers with two different IDENTITY_KEY constants — no live server needed. For TEST-06.9 (live isolation), require `BSV_TEST_ROOT_KEY_2` env var, gate with `#[ignore]`, skip if not set.
