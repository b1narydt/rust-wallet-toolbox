<objective>
Define the WalletStorageProvider async trait with ~25 methods matching the TS WalletStorageProvider interface hierarchy, implement a blanket impl over StorageProvider so SqliteStorage satisfies it automatically, and add is_storage_provider() with a default true return.

Purpose: This trait is the contract that StorageClient (Phase 3) will implement directly and that WalletStorageManager (Phase 4) will accept as `Arc<dyn WalletStorageProvider>`. Without it, there is no type-safe abstraction for remote vs local storage.

Output: `wallet_provider.rs` with trait + blanket impl, supporting types (`RequestSyncChunkArgs`, extended `AuthId`), and compile/runtime tests.
</objective>

  **Step 3: Create WalletStorageProvider trait + blanket impl** in new file `src/storage/traits/wallet_provider.rs`.

  The trait must be `#[async_trait]` and `Send + Sync` for object safety. Declare all 25 methods from the TS interface hierarchy as documented in 02-RESEARCH.md "The 25 Rust Method Signatures" section. Key design decisions:

  - `is_storage_provider(&self) -> bool` has default body returning `true`. StorageClient (Phase 3) overrides to `false`.
  - `is_available(&self) -> bool` and `get_settings(&self) -> WalletResult<Settings>` are sync (non-async). In the blanket impl, `get_settings` delegates to `StorageProvider::get_settings(self, None).await` -- BUT since it needs to be sync for the TS parity, make it async in the trait too. The research note about sync is aspirational; the TS `getSettings` returns a cached value but we can start async and optimize later. So: `async fn get_settings(&self) -> WalletResult<Settings>`.
  - `is_available` stays sync (no `async`).
  - For methods that take `auth: &AuthId` in WalletStorageProvider but the underlying StorageProvider/Reader/Writer uses `auth: &str`, the blanket impl extracts `auth.identity_key.as_str()`.
  - For `list_actions`, `list_certificates`, `list_outputs`: these use bsv-sdk wallet types (`bsv::wallet::ListActionsArgs`, etc.). Import them. The blanket impl should return `WalletError::NotImplemented` for these since they are wallet-level operations not directly in StorageReader/Writer. StorageClient (Phase 3) will provide the real implementations via RPC.
  - For `abort_action`, `relinquish_certificate`, `relinquish_output`: same pattern -- blanket impl returns `NotImplemented`, StorageClient provides real impl.
  - For `create_action`, `process_action`, `internalize_action`: the blanket impl CAN delegate to `StorageActionProvider` methods. But StorageActionProvider takes `auth: &str` and different arg types. The WalletStorageProvider versions should use their own arg types matching TS. For the blanket impl, delegate through: extract `auth.identity_key`, call `StorageActionProvider::create_action(self, &auth.identity_key, args, None)`. However, the action types overlap but are not identical. If the types mismatch, return `NotImplemented` in the blanket impl and document that the local path uses `StorageActionProvider` directly, not through `WalletStorageProvider`.
  - For `get_sync_chunk` and `process_sync_chunk`: these take `RequestSyncChunkArgs` (the new wire-format struct). The blanket impl must convert `RequestSyncChunkArgs` to the internal `GetSyncChunkArgs<'a>` and call the existing sync functions. This requires building a `SyncMap` from the offsets. If the conversion is complex, return `NotImplemented` in the blanket impl for now -- the local path calls `get_sync_chunk()` and `process_sync_chunk()` directly via the sync module, not through WalletStorageProvider.
  - For `find_or_insert_sync_state_auth`: delegates to `StorageReaderWriter::find_or_insert_sync_state(self, &auth.identity_key, storage_identity_key, storage_name, None)`.
  - For `set_active` (the async WalletStorageProvider version taking AuthId): this is different from the sync `StorageProvider::set_active(bool)`. The TS version calls `setActive(auth, newActiveStorageIdentityKey)` which updates the user's active storage. Blanket impl: return `NotImplemented`.
  - For `get_services` and `set_services`: include with default `NotImplemented` body (no blanket impl override needed).
  - For `migrate`: delegates to `StorageProvider::migrate_database(self)`.

  **Pragmatic approach for blanket impl:** Methods that have direct equivalents in StorageProvider/Reader/Writer should delegate. Methods that are wallet-level (list_actions, abort_action, relinquish_*, set_active-with-auth) or need complex type conversion should return `WalletError::NotImplemented` in the blanket impl. The blanket impl's primary purpose is to let SqlxProvider satisfy the trait so it can be used as `Arc<dyn WalletStorageProvider>` -- the local code path does NOT call these wallet-level methods through WalletStorageProvider. StorageClient (Phase 3) provides real implementations for ALL methods via RPC.

  Check if `WalletError::NotImplemented` variant exists. If not, check what error variant is appropriate (e.g., `WalletError::Internal("not implemented".into())`). Use whatever error pattern the codebase already uses for unimplemented paths.
