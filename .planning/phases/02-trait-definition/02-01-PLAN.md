<objective>
Define the WalletStorageProvider async trait with ~25 methods matching the TS WalletStorageProvider interface hierarchy, implement a blanket impl over StorageProvider so SqliteStorage satisfies it automatically, and add is_storage_provider() with a default true return.

Purpose: This trait is the contract that StorageClient (Phase 3) will implement directly and that WalletStorageManager (Phase 4) will accept as `Arc<dyn WalletStorageProvider>`. Without it, there is no type-safe abstraction for remote vs local storage.

Output: `wallet_provider.rs` with trait + blanket impl, supporting types (`RequestSyncChunkArgs`, extended `AuthId`), and compile/runtime tests.
</objective>

  **Step 3: Create WalletStorageProvider trait + blanket impl** in new file `src/storage/traits/wallet_provider.rs`.

  The trait must be `#[async_trait]` and `Send + Sync` for object safety. Declare all 25 methods from the TS interface hierarchy as documented in 02-RESEARCH.md "The 25 Rust Method Signatures" section. Key design decisions:

  - `is_storage_provider(&self) -> bool` has default body returning `true`. StorageClient (Phase 3) overrides to `false`.
  - `is_available(&self) -> bool` and `get_settings(&self) -> WalletResult<Settings>` are sync (non-async). In the blanket impl, `get_settings` delegates to `StorageProvider::get_settings(self, None).await` -- BUT since it needs to be sync for the TS parity, make it async in the trait too. The research note about sync is aspirational; the TS `getSettings` returns a cached value but we can start async and optimize later. So: `async fn get_settings(&self) -> WalletResult<Settings>`.
  - `is_available` stays sync (no `async`).
  - For methods that take `auth: &AuthId` in WalletStorageProvider but the underlying StorageProvider/Reader/Writer uses `auth: &str`, the blanket impl extracts `auth.identity_key.as_str()`.
